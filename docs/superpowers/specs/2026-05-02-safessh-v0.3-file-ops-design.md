## 1. Goal

Ship `safessh <project> read <path>` and `safessh <project> write <path>` as first-class, policy-gated, audit-logged file operations over SFTP. Add path-glob-aware policy rules so projects can permit reads under `/etc/nginx/*` without permitting reads anywhere. Audit every read/write with size and pre-redaction sha256.

Out of scope (deferred to later milestones): directory listing, recursive copy, `chmod`/`chown`, partial-range reads, resume, second-binary fallback (e.g. `rsync`).

## 2. Non-goals — and why

- **Directory tree walking.** Adds policy surface (does the glob apply to each entry?) and surprise costs (recursive listing of `/`). Out.
- **Range reads.** Avoid until a real use case shows up; otherwise we ship a feature that complicates redaction (you can't redact across a chunk boundary you don't see).
- **Write-stream-from-file flag.** A `--from-file <local>` option lets an agent silently exfiltrate by pointing at the wrong path. Stdin only.
- **Rewriting `exec` to share the file path.** `read` and `write` are first-class operations, not synthesized `cat`/`tee` commands. Spec §4.5 already names them as separate categories — this spec honors that.

## 3. User-facing surface

```
safessh <project> [--on <target>] read  <remote-path>            # bytes → stdout (framed)
safessh <project> [--on <target>] write <remote-path> < local    # stdin → remote, atomic
```

Both inherit `--yolo`, `--on`, project resolution, the framing protocol, exit codes, and audit guarantees from `exec`.

### 3.1 Read

- Streams remote bytes through the redactor, then to stdout, wrapped in `<stdout>...</stdout>` framing.
- Size cap: `output.file_read_cap_bytes` (default `5_242_880`). On exceed: emit cap-many bytes, exit 30, set `truncated: true` in audit.
- File not found: exit 1, stderr `safessh: error: no such remote file: <path>`. (Permission-denied / other sftp errors: exit 20.)
- Outputs nothing visible until the audit `file_read` event has been written (SAFETY-INVARIANT-4).

### 3.2 Write

- Reads stdin until EOF or cap, writes to a remote temp path under the same directory, then sftp-renames over the target.
- Size cap: `output.file_write_cap_bytes` (default `5_242_880`). On exceed: abort upload, delete temp, exit 30 with `truncated: true` in audit, do not touch the target.
- Refuses if the parent directory doesn't exist: exit 1, `no such remote directory: <dir>`. (Won't auto-`mkdir` — that's a different operation, deferred.)
- Refuses if target exists and is a directory: exit 1, `target is a directory: <path>`.
- Symlink target paths: resolved via `realpath` before glob matching; written through to whatever `realpath` resolves (no symlink-create magic, no symlink-follow refusal).
- No redaction on the upload path. Rationale: bytes originate with the agent — redaction prevents server-secret leakage *to* the agent, not the reverse. Redacting silently would corrupt user input.

### 3.3 Headless approval

Same `BLOCKED:<token>` format as `exec`. The token resolves via `safessh approve <token>` and the rule it produces (timed/always/blocked) gains a path component:

- For exec-style rules: unchanged.
- For file rules: when the user picks "Always allow", the new rule lands in `[[policy.file_rules]]` with `paths = [<exact path that triggered the prompt>]`. The user can broaden via TUI later.

## 4. Schema changes

### 4.1 New `[[policy.file_rules]]` table

Backward compatible: v0.2 projects parse unchanged because the field is optional and defaults to empty.

```toml
[policy]
allow = ["read:safe", "file:read"]
require_approval = ["file:write", "destructive:filesystem"]
deny = ["destructive:disk"]

[[policy.file_rules]]
category = "file:read"      # "file:read" | "file:write"
paths    = ["/etc/nginx/*", "/var/log/nginx/*"]
decision = "allow"          # "allow" | "approve" | "deny" | "block"

[[policy.file_rules]]
category = "file:write"
paths    = ["/tmp/safessh-staging/*"]
decision = "allow"
```

Path syntax: standard `globset` (`*` single-segment, `**` recursive, `?`, `[abc]`, no shell brace expansion). Globs match against the **canonical** path returned by remote `realpath`, after client-side `~` and `..` expansion.

### 4.2 Decision precedence

`decide()` extended (semantics — code lives in `safessh-policy/src/decision.rs`):

1. Block-list pattern match → `Block`. (SAFETY-INVARIANT-2 unchanged.)
2. Matched `file_rules` entry, in order: `decision = "block"` → `Block`; `"deny"` → `Deny`; `"approve"` → `RequireApproval`; `"allow"` → `Allow`.
3. Project policy `deny` covers any matched category → `Deny`.
4. Unexpired timed-rule pattern → `Allow`.
5. Always-rule pattern → `Allow`.
6. All matched categories in `policy.allow`, none in `require_approval` → `Allow`.
7. Otherwise → `RequireApproval`.

The `file_rules` block runs **before** category-level allow/require_approval but **after** the global block-list. This preserves SAFETY-INVARIANT-2 ("deny > allow ordering") for both rule kinds.

### 4.3 Shipped preset additions

`policies/presets.toml` (read-only, binary-shipped) gains a default deny-list of sensitive files for both `file:read` and `file:write`:

```toml
[[file_rules]]
category = "file:read"
paths    = ["/etc/shadow", "/etc/sudoers", "/etc/sudoers.d/**",
            "/root/.ssh/**", "/home/*/.ssh/**", "**/.env*",
            "**/id_rsa*", "**/id_ed25519*", "**/id_ecdsa*"]
decision = "deny"

[[file_rules]]
category = "file:write"
paths    = ["/etc/shadow", "/etc/sudoers", "/etc/sudoers.d/**",
            "/root/.ssh/**", "/home/*/.ssh/**"]
decision = "deny"
```

These apply implicitly to every project (the preset is merged in at decision time, not copied into per-project TOML). Users can't bypass by writing a more permissive `allow` — preset denies are evaluated before per-project rules. (New SAFETY-INVARIANT-14, see §7.)

### 4.4 Audit schema

Two new event types, `schema_version: 1` (no migration — `data` is `serde_json::Value`):

`file_read`:
```json
{
  "event_type": "file_read",
  "project": "prod",
  "data": {
    "target": "web",
    "path": "/etc/nginx/nginx.conf",
    "canonical_path": "/etc/nginx/nginx.conf",
    "decision": "allow",
    "matched_rule": "file_rules[0]",
    "categories": ["file:read"]
  }
}
```

`file_read_complete`:
```json
{
  "event_type": "file_read_complete",
  "project": "prod",
  "data": {
    "target": "web",
    "path": "/etc/nginx/nginx.conf",
    "bytes_returned": 4823,
    "sha256": "<hex of pre-redaction bytes>",
    "redactions": [{"type": "aws_access_key", "count": 0}],
    "truncated": false,
    "duration_ms": 142
  }
}
```

`file_write` / `file_write_complete` mirror these. The `_complete` event records bytes-as-written sha256 and `truncated` (true only if stdin exceeded the cap and the upload was aborted; in that case `bytes_written: 0` because the rename never happened).

## 5. Components

No new crates. Changes by crate:

### 5.1 `safessh-core`

- No new types. `AuditEvent.data` already accepts arbitrary JSON.
- `Redactor` already operates on `&[u8]` — no changes.

### 5.2 `safessh-storage`

- `Policy` struct (`crates/safessh-storage/src/project.rs`) gains `pub file_rules: Vec<FileRule>` with `#[serde(default)]`.
- New `FileRule { category: String, paths: Vec<String>, decision: FileDecision }` with `FileDecision = Allow | Approve | Deny | Block`.
- New `policies::presets` module exposing the binary-embedded preset deny-list (deserialized once from `include_str!("presets.toml")`).
- New paths: none (file content moves through SSH driver, not via filesystem).

### 5.3 `safessh-policy`

- `decision.rs` extended: after block-list, before category-level rules, evaluate `file_rules` (preset + project, preset evaluated first). Pure function — no new I/O.
- New `FileOp { Read, Write }` discriminator on `DecisionInput` so the engine knows which `category` to target. (For exec, `FileOp` is `None`.)
- New `match_file_rules(op, path, rules)` helper. Glob compilation cached per call (one `globset::GlobSet` build per decide()).

### 5.4 `safessh-ssh`

`SshDriver` trait gains two methods:

```rust
async fn read_file(
    &self,
    target: &Target,
    path: &str,
    cap_bytes: u64,
) -> Result<FileReadResult>;

async fn write_file(
    &self,
    target: &Target,
    path: &str,
    bytes: &[u8],
) -> Result<FileWriteResult>;

pub struct FileReadResult { pub bytes: Vec<u8>, pub canonical_path: String, pub truncated: bool }
pub struct FileWriteResult { pub canonical_path: String, pub bytes_written: u64 }
```

`OpenSshDriver` implementation (`crates/safessh-ssh/src/openssh.rs`):
- Drives `sftp -b - <target>` over the existing ControlMaster socket. Batch script for read: `get -p <path> /dev/stdout` plus `realpath <path>`. For write: `put` to a `.safessh.<random>.tmp` sibling, then `rename` to target.
- Parses sftp's framed status output to detect "no such file" / "permission denied" / "is a directory".
- Cap enforcement: on read, monitor stdout byte count, kill the sftp child when the cap is exceeded (truncated). On write, count stdin bytes before transmit, refuse if over cap (don't even start transfer).
- Temp name: `.safessh.<8 hex chars>.tmp`. Cleanup on any failure path (`rm` via the same batch).

`MockSshDriver` (`crates/safessh-ssh/src/mock.rs`) gets an in-memory `BTreeMap<(target_id, path), Vec<u8>>` for tests. Mock honors caps and surfaces `NotFound`/`IsDirectory` errors deterministically.

### 5.5 `safessh-cli`

- Two new subcommands `read` / `write` per project, parallel to `exec`. Argv parsing, framing, redaction, audit dispatch all mirror `exec.rs` — pull shared bits into `commands/file_common.rs` rather than duplicate.
- `commands/file_common.rs` holds the decide → audit `_attempt` → driver call → audit `_complete` flow. `exec.rs` does **not** need to change (different operation), but the audit-write-before-output invariant is enforced by the same helper signature both call sites use.

### 5.6 `safessh-tui`

- Approvals screen learns one new row variant: `file:read /path` / `file:write /path` instead of `<binary> <flags>`. Picker actions for "Always allow" write a `file_rules` entry instead of a `PatternRule`.
- Rules screen gains a fourth tab: `[F]ile rules`. Lists per-project, lets user delete entries.
- Audit screen one-line summary: `file_read /etc/nginx/nginx.conf (4.8KB)` / `file_write /tmp/foo (12B, truncated)`.
- Help overlay: add `r` (read), `w` (write) — actually no, those are CLI verbs, not TUI keys. Just document the new approval/rule shapes.

### 5.7 `safessh-skill`

- Embedded `safessh.md` gains a "Reading and writing files" section: when to use `read`/`write` vs `exec "cat …"`, redaction caveats on read, atomicity guarantee on write, the `BLOCKED:<token>` flow for path approvals.
- Insta snapshots regenerated.

### 5.8 `safessh-audit`

- No code change required (the schema is just new event-type strings in `data`). The SQLite indexer (which doesn't ship until v0.5) will need a migration to surface `file_*` events as a typed view — out of scope for v0.3.

## 6. Data flow

### 6.1 Read

```
LLM: safessh prod read /etc/nginx/nginx.conf
  ├─[cli]────▶ resolve project + target
  ├─[ssh]────▶ realpath /etc/nginx/nginx.conf  (sftp batch)
  ├─[policy]─▶ decide(FileOp::Read, "/etc/nginx/nginx.conf",
  │                    project_policy, preset_file_rules + project_file_rules,
  │                    timed, always, blocks)
  ├─[audit]──▶ {event: file_read, decision: allow, matched_rule: ...}
  ├─[ssh]────▶ get -p /etc/nginx/nginx.conf  (cap enforced byte-by-byte)
  ├─[core]───▶ Redactor.scan(&bytes) → counts per type
  ├─[cli]────▶ <stdout>...redacted bytes...</stdout>  <exit code="0" .../>
  └─[audit]──▶ {event: file_read_complete, sha256, redactions, truncated:false}
```

### 6.2 Write

```
LLM: safessh prod write /tmp/staging/x.conf < local-file
  ├─[cli]────▶ read stdin → buffer (refuse if > cap; stream to disk-backed tempfile if needed)
  ├─[ssh]────▶ realpath /tmp/staging/x.conf  (or parent if target absent)
  ├─[policy]─▶ decide(FileOp::Write, "/tmp/staging/x.conf", ...)
  ├─[audit]──▶ {event: file_write, decision: allow, ...}
  ├─[ssh]────▶ put → /tmp/staging/.safessh.<rand>.tmp
  ├─[ssh]────▶ rename -> /tmp/staging/x.conf
  ├─[cli]────▶ <stdout/> <exit code="0" .../>
  └─[audit]──▶ {event: file_write_complete, sha256, bytes_written: N}
```

Failure mid-transfer: `rename` never runs, `rm` cleans up the temp via the same sftp batch; `file_write_complete` records `bytes_written: 0, error_class: "ssh_failure"`.

## 7. Safety invariants (additions)

Existing invariants 1–12 unchanged. Two additions:

- **SAFETY-INVARIANT-13: Atomic remote writes.** Every `write_file` uploads to a sibling tempfile and atomically renames. No half-written remote file observable. Marker on `OpenSshDriver::write_file`.
- **SAFETY-INVARIANT-14: Preset file_rules evaluated before project file_rules.** A user-supplied permissive `allow` cannot override the shipped sensitive-path deny-list. Marker on `decide::match_file_rules`.

## 8. Error model

Reuses existing exit codes:

- `0` success
- `1` not found (file or project), is-a-directory, parent-dir missing
- `2` usage error (e.g. `read` with no path)
- `10` approval required (BLOCKED token, same format as exec)
- `11` blocked by rule
- `12` denied
- `20` sftp failure (permission denied, network error, sftp protocol error)
- `21` connection refused / timeout
- `30` exceeded cap (read or write — `truncated: true` in audit)
- `40` storage error (project TOML corrupt, lock held)
- `50` audit-write failure

No new exit codes. The skill markdown gets an updated "Exit codes for read/write" subsection mapping these to LLM-facing guidance.

## 9. Testing strategy

- **Unit (pure):** `decide()` with file_rules — preset > project precedence, deny > allow, no-rule fallback. Glob matching boundary cases (`*` vs `**`, `~` expansion, symlinks resolved via mock realpath).
- **Unit (storage):** v0.2 project TOML round-trips unchanged (no `file_rules` field). v0.3 project with `file_rules` round-trips; bad `decision` enum value fails parse with helpful error.
- **Driver (mock):** read truncation at cap, write rejection over cap, atomic write semantics (failure leaves original), is-a-directory error class.
- **Integration (Linux container, `--features integration`):** real sftp against an OpenSSH server in a Docker container. Read /etc/hostname, write to /tmp, attempt write to /etc/shadow → blocked by preset.
- **Audit:** every read/write produces both `_attempt` and `_complete` events. sha256 in `_complete` matches the bytes returned (read) or written (write).
- **Skill:** insta snapshot of updated `safessh.md`.
- **TUI:** new approval row variant renders; new file-rules tab in rules screen renders empty + non-empty.

## 11. Migration

None. v0.2 project TOMLs parse unchanged because `file_rules` is `#[serde(default)]`. The preset deny-list is a new tightening — agents that previously could `exec "cat /etc/shadow"` in `--yolo` could still do that (yolo bypasses policy by design); but the preset stops the new first-class `read /etc/shadow` path even on a misconfigured project. That's the correct behavior per SAFETY-INVARIANT-9 (yolo doesn't bypass audit, but it does bypass policy — the preset is policy, so a yolo'd `read` would still go through; this is documented in the skill).

## 12. Open questions

- **`ProxyJump` over sftp.** Spec §6.4 doesn't address it. ssh-config's `ProxyJump` works through OpenSSH's sftp invocation transparently — confirm during integration testing.
- **Performance budget.** Reading a 5 MB file over sftp on a slow link can take several seconds. v0.7's perf budget will set the regression target; for v0.3 we just measure and document baseline.
