# safessh v0.5.0 + v0.6.0 — Design

**Status:** Draft, ready for review.
**Authors:** safessh maintainers.
**Scope:** Implements milestones v0.5.0 (audit power) and v0.6.0 (multi-agent skill targets) from the [v1 design spec §9](2026-04-30-safessh-design.md#9-milestones).

This document is the source of truth for these two milestones. The implementation plan that consumes this spec lives at `docs/superpowers/plans/2026-05-03-safessh-v0.5-v0.6.md`.

---

## 1. Goal

Two shippable, independent improvements:

- **v0.5.0 — Audit power.** Make the audit log usefully queryable. Add a SQLite index built lazily from the canonical JSONL, expose structured filters via `safessh audit query`, and lift the TUI audit screen's 200-event tail cap so users can browse history with the same filter set. JSONL remains authoritative.
- **v0.6.0 — Multi-agent skill targets.** Add four new skill adapters (`cursor`, `gemini-cli`, `codex`, `plain`) so safessh can install its agent skill for whichever framework the user runs. Add `safessh skill update` (re-render and reinstall to detected locations) and `--target all` (install to every supported (target, scope) combo).

Both milestones land **without changing any safety invariant**. The SQLite index is a derived cache; the JSONL is still the source of truth. The new adapters reuse the same canonical skill body in `safessh-skill/src/content/safessh.md`.

## 2. Non-goals

- **No daemon, no background indexer process.** The index is rebuilt lazily on read. Background-watcher designs are explicitly deferred (post-v1).
- **No first-class structured DB queries from the LLM.** `audit query` stays the LLM-facing surface; the LLM never opens SQLite directly. Structured DB access is a P3 deferred feature in §9.1 of the v1 spec.
- **No new audit event types.** The schema_version stays at `1`. v0.5 indexes what already gets written.
- **No skill content rewrite for v0.6.** The four new adapters wrap the same canonical body that v0.1's `claude-code` and `agents-md` adapters already use. Per-target tweaks are limited to header/frontmatter/section wrapping.
- **No telemetry or auto-update from `safessh skill update`.** It re-renders the embedded body and rewrites locally-installed copies. No network calls (SAFETY-INVARIANT-6).

## 3. v0.5.0 — Audit power

### 3.1 Architecture

```
                  append (existing)
JSONL log  ◀────────────────────────  safessh-audit::AuditWriter
   │                                    (unchanged: SAFETY-INVARIANT-4)
   │ read tail (existing tail: TUI)
   │
   ▼
┌──────────────────────────────────────────┐
│            safessh-audit (new)           │
│                                          │
│   sqlite::Index   ◀───  catch_up()       │
│      │                  reads JSONL      │
│      │                  from offset      │
│      │                  → INSERT rows    │
│      ▼                                   │
│   query::Query        ◀── filter args    │
│      │                                   │
└──────┼───────────────────────────────────┘
       ▼
   raw_json (string) ─▶ table | jsonl | count formatter (CLI)
```

- **Crate boundaries unchanged** — `safessh-audit` already owns audit I/O. It gains a `sqlite` submodule and a `query` submodule. It does not gain new outbound interfaces (no calls into `safessh-cli`, no new traits across crates).
- **`safessh-tui` audit screen** moves from "in-memory ring buffer of last 200 events" to "open the index, fetch a windowed query, plus tail new events live as JSONL appends arrive." Live tailing logic stays; the SQLite call replaces the bounded ring.
- **Filesystem additions:** one new file, `~/.local/state/safessh/audit.db` (Linux) / `~/Library/Application Support/safessh/audit.db` (macOS), beside `audit.log`.

The split between v0.5 and v0.4 architectures is small: one new module group inside an existing crate, plus a swap-in for one TUI screen. No new crates.

### 3.2 SQLite index

#### 3.2.1 Schema

```sql
-- v1
CREATE TABLE events (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    byte_offset     INTEGER NOT NULL,        -- offset of this event's line in audit.log
    timestamp       TEXT    NOT NULL,        -- RFC3339 UTC, sortable lexicographically
    event_type      TEXT    NOT NULL,
    project         TEXT,
    target          TEXT,                    -- nullable; populated when present in event.data
    decision        TEXT,                    -- allow | require_approval | deny | block
    exit_code       INTEGER,                 -- nullable; from exec_complete events
    raw_json        TEXT    NOT NULL         -- full event line, for fall-through queries
);

CREATE INDEX idx_events_timestamp           ON events(timestamp);
CREATE INDEX idx_events_project_timestamp   ON events(project, timestamp);
CREATE INDEX idx_events_event_type_timestamp ON events(event_type, timestamp);
CREATE INDEX idx_events_decision_timestamp  ON events(decision, timestamp);

CREATE TABLE meta (
    key   TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
-- Required keys:
--   schema_version       : "1"
--   last_indexed_offset  : "<bytes>" (where the next catch-up should resume)
--   source_file          : absolute path of the JSONL the index was built from
```

**Why narrow + `raw_json`:** the structured filters chosen for v0.5 (`--project`, `--type`, `--target`, `--decision`, `--exit-code`, `--since`, `--until`, `--limit`) all hit columns the table already has. Anything we add later (file path searches, sha256 lookups) can use `raw_json` with `LIKE` or SQLite's JSON1 extension without a schema change. The cost: lookups by, say, file path are slower until/unless we add a column. That's a deliberate trade — schema churn is more painful than a 50ms query in v0.5.

#### 3.2.2 Index lifecycle

```
┌──────────────────────────────────────────┐
│  open_or_create(paths) -> Index          │
│   1. open `audit.db`, run pending        │
│      refinery migrations.                │
│   2. read meta.source_file. If unset,    │
│      set to current paths.audit_log().   │
│      If set and != current path → drop   │
│      events, reset offset to 0.          │
└──────────────────┬───────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────┐
│  catch_up(idx)                           │
│   1. stat audit.log; if size < offset,   │
│      reset offset to 0 (rotation).       │
│   2. seek to offset, read remaining      │
│      lines.                              │
│   3. begin transaction. for each line:   │
│      parse, extract columns, INSERT.     │
│   4. commit, update last_indexed_offset. │
└──────────────────┬───────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────┐
│  query(idx, filters) -> rows             │
│   build WHERE clause from filters,       │
│   ORDER BY timestamp DESC, LIMIT N.      │
└──────────────────────────────────────────┘
```

- **Lazy catch-up.** Triggered by `Index::query` and by the TUI audit screen's reload paths (initial open + `FsEvent::AuditAppended`). Never triggered by `safessh ... exec` or any write-path command — exec stays JSONL-only on the hot path. (See §3.4 for the failure model.)
- **Rotation handling.** When `AuditWriter` rotates `audit.log` to `audit.log.<ts>` at 100 MiB, the next `catch_up` sees `metadata.len() < last_indexed_offset` and rebuilds from offset 0 against the new file. Rotated files are never indexed (they're frozen history; `audit query --grep` over the live log + rotated files via filesystem fallback is acceptable; full historical indexing across rotations is post-v0.5).
- **Transactions.** Catch-up wraps inserts in a single transaction (per call). If insertion fails midway, the transaction rolls back; `last_indexed_offset` is unchanged; next catch-up retries from the same point.
- **Schema migrations** via `refinery`. Migrations live under `safessh-audit/src/sqlite/migrations/`. v0.5 ships migration `V1__initial.sql`. A `schema_version` mismatch (e.g., older binary opening a newer DB) triggers a hard error: "audit index built by newer safessh; ignoring index, JSONL is unaffected." The CLI proceeds with empty SQLite results and a stderr note.
- **Corruption recovery.** If SQLite returns `SQLITE_CORRUPT` or migrations fail unrecoverably, drop the file, recreate it, reset `last_indexed_offset` to 0. The JSONL is unaffected. This is a logged warning, not a fatal error.

#### 3.2.3 Column extraction rules

For each parsed JSONL line, populate columns from these field paths (all optional except `event_type` and `timestamp`):

| Column        | Source                                                     |
|---------------|------------------------------------------------------------|
| `timestamp`   | `timestamp` (top-level)                                    |
| `event_type`  | `event_type` (top-level)                                   |
| `project`     | `project` (top-level)                                      |
| `target`      | `data.target` if present (file ops, tunnels)               |
| `decision`    | `data.decision` if present (exec_attempt, file_attempt)    |
| `exit_code`   | `data.exit_code` if present (exec_complete)                |
| `raw_json`    | the entire line, verbatim                                  |
| `byte_offset` | byte offset of the line's first character in the source file|

A line that fails JSON parsing is **skipped, not failed** — the line stays in JSONL (authoritative), and the index just doesn't see it. Skipped-line count is logged at debug level. Catch-up still advances `last_indexed_offset` past unparseable lines so we don't re-attempt forever.

#### 3.2.4 Additive event field: `data.target` on exec events

`exec_attempt` and `exec_complete` events do **not** currently carry `data.target` (only `file_*` and `tunnel_*` events do). Without that field, `audit query --target prod` would silently miss exec events — broken UX. v0.5 extends the two exec constructors in `safessh-audit/src/event.rs` to populate `data.target` when the call site has it (the CLI already resolves the target at exec time).

This is an **additive** change — existing JSONL lines without the field still parse fine; `data.target` is JSON-optional. `schema_version` does **not** bump (the change is "field added to existing event types," not a structural change). Older safessh binaries reading newer JSONL still work. The same applies for any other event type that resolves a target at the call site (only exec is affected today).

Updated extraction in §3.2.3 needs no change — `target` column already pulls from `data.target`.

### 3.3 `safessh audit query` CLI

**Existing flags (kept):**
- `--project <name>`
- `--type <event_type>`
- `--grep <pattern>`

**New flags (v0.5):**
- `--since <RFC3339|duration>`     — e.g. `2026-05-01T00:00:00Z` or `7d`, `24h`, `30m`
- `--until <RFC3339|duration>`     — same shape; relative durations are anchored to now
- `--limit N`                      — default 100; `--limit 0` means unlimited
- `--decision <allow|require_approval|deny|block>`
- `--exit-code <N|N..M>`           — exact code (e.g., `--exit-code 50`) or inclusive range (`--exit-code 1..255`)
- `--target <name>`                — target host slot (e.g., `prod`)
- `--format <jsonl|table|count>`   — default `jsonl` (LLM-facing), `table` is humans, `count` prints just `N\n`

**Behavior:**
- All flags AND together. Multiple values for the same flag are not supported in v0.5 (keep the surface small).
- Default ordering is `ORDER BY timestamp DESC` (newest first), matching how humans read logs. JSONL output remains valid newline-delimited JSON.
- `--grep <pattern>` is applied **after** the SQL filter, against `raw_json`, with `--grep` matching as a substring (no regex). Matches existing v0.4 semantics.
- Exit codes: `0` on success (any number of rows including zero); `30` on bad flag combinations (e.g., `--since` after `--until`).

**Exit-code range parser:** `N` → `code = N`. `N..M` → `code BETWEEN N AND M`. Anything else → exit 30 with `safessh: usage: --exit-code expects N or N..M`.

**Duration parser:** matches `^(\d+)(s|m|h|d|w)$`. Anything else falls through to RFC3339 parse. Both fail → exit 30.

### 3.4 Failure model and the JSONL guarantee

These are non-negotiable per the project's safety invariants ([CLAUDE.md §load-bearing rules](../../../CLAUDE.md), spec §7.2):

- **Exec / file / tunnel paths never touch SQLite.** They write JSONL, fsync, return. If SQLite is missing, broken, or stale, exec is unaffected. This is enforced by keeping all SQLite calls inside `audit::sqlite::*` and `audit::query::*`, with no callers from `safessh-cli/src/commands/{exec,read,write,forward}.rs`.
- **`audit query` degrades gracefully.** If opening the DB fails, the command falls back to the existing JSONL grep path automatically and prints a `safessh: warning: audit index unavailable, falling back to log scan` to stderr. Exit status: 0 if results found, 0 if none, never the SQLite error (it's a warning).
- **TUI audit screen also degrades.** If catch-up errors out, the screen displays the existing in-memory tail + a status-line message. New events still arrive via the existing FS watcher.
- **The CLAUDE.md rule "JSONL is authoritative; SQLite can fail silently" is preserved verbatim.** The fallbacks above are the codification of that rule.

### 3.5 TUI audit screen changes

Existing screen (lib.rs::screens::audit, ~466 lines) currently keeps an in-memory ring of 200 events plus a live-tail watcher. v0.5 makes these targeted changes:

- **Replace ring buffer with windowed SQL query.** On open and on `FsEvent::AuditAppended`, run `Index::catch_up` then `Index::query(filters, limit=200)`. Display rows. (The TUI window of 200 is independent of the CLI `--limit 100` default — different surfaces, different defaults; both are tunable inside their own context.)
- **Live tail still works.** When new lines arrive in JSONL while the screen is open, catch-up indexes them and the screen re-runs the windowed query, picking up new rows automatically.
- **Add filter fields:** `since`, `until`, `decision`, `exit_code`, `target` — same edit-buffer pattern as the existing project/type/grep fields.
- **Show total count.** Status line shows `n shown / N total matches` to make the windowing visible.
- **Pagination.** `Page Up` / `Page Down` shifts the window through results; reaching the top auto-resumes live tail. (Existing `j`/`k` selection still works within the page.)
- **Fallback.** If SQLite is unavailable, the screen falls back to the existing JSONL-tail behavior unchanged. Status line shows `(index unavailable: log-scan mode)`.

Existing snapshot tests in `safessh-tui/tests/snapshots/` must continue to pass, with one new snapshot for the SQLite-backed view and one for the fallback-mode banner.

### 3.6 Documentation deliverable: `docs/audit.md`

Lands as part of v0.5. Contents:

1. **JSONL schema reference.** One subsection per event_type (12 currently). Per type: `data.*` fields, types, examples. Lifted from the existing event constructors in `safessh-audit/src/event.rs`.
2. **SQLite schema reference.** The DDL above, plus example `sqlite3` recipes (raw queries are not officially supported, but power users will run them).
3. **`audit query` examples.** Common filter combinations with expected output. Includes the LLM-oriented "use this from the skill" pattern.
4. **Recovery recipes.** `rm audit.db && safessh audit query` to rebuild. What to look for in the warning lines.

This is also the artifact that the skill markdown points readers at, so it must be accurate.

### 3.7 Testing strategy (v0.5)

- **Unit, fast tier (in `crates/safessh-audit/tests/`):**
  - Round-trip: write 1 / 100 / 10k events to a temp JSONL, run `catch_up`, query, assert filter correctness.
  - Each filter dimension: `--since/--until` boundary cases, `--exit-code` range parsing, `--decision` exact match, `--target` matches `data.target` not `target` columns elsewhere.
  - Rotation: write to `audit.log`, index it, rename to `audit.log.<ts>`, write a new `audit.log`, index again — assert the new file is reindexed from offset 0 and old entries are gone (this is documented behavior, not a bug).
  - Schema mismatch: write `meta.schema_version = 99`, open the index, assert hard-error path.
  - Corruption: truncate the SQLite file mid-page, open the index, assert recovery (drop + rebuild).
- **Property tier (`proptest`):**
  - Random sequences of valid `AuditEvent` JSON → catch-up + query reproduces filter results matching a brute-force JSONL scan.
- **Integration tier (gated, runs in container CI on Linux):**
  - End-to-end: run a series of `exec` / `file read` / `tunnel open` operations, then `audit query` with mixed filters, compare to grep over the JSONL.
- **TUI snapshot tier (`insta`):**
  - SQLite-backed audit screen with a fixture DB.
  - Fallback-mode audit screen when the DB is removed mid-session.
- **Bench (informational):**
  - Catch-up of a 50k-event JSONL completes < 1.5s on M1 baseline. Recorded in `crates/safessh-audit/benches/`. Not a CI gate in v0.5; reference for v0.9 perf budget.

### 3.8 Rollout / migration

There is no on-disk DB before v0.5, so first run on an existing install does a single full catch-up of whatever's accumulated in `audit.log`. Worst case: ~1.5s for the largest log we'd expect pre-v0.5 (under 10 MiB, since rotation kicks in at 100 MiB and most users won't be near that). If catch-up exceeds 5s, the CLI prints `safessh: indexing audit log…` to stderr once.

## 4. v0.6.0 — Multi-agent skill targets

### 4.1 Architecture

The skill crate `safessh-skill` is already structured around the right abstraction: `adapters::Target` enum, `format(target, body) -> String`, `default_path(target, scope, cwd) -> Option<PathBuf>`, plus `detection::detect`. v0.6 grows the enum from 2 variants to 6 and adds the corresponding format/path/detection rows.

```
Target::ClaudeCode  ── format ──▶ frontmatter + body (existing)
Target::AgentsMd    ── format ──▶ "## safessh" section (existing)
Target::Cursor      ── format ──▶ cursor frontmatter + body                   ◀── NEW
Target::GeminiCli   ── format ──▶ "## safessh" section in GEMINI.md           ◀── NEW
Target::Codex       ── format ──▶ "## safessh" section in ~/.codex/AGENTS.md  ◀── NEW
Target::Plain       ── format ──▶ raw body, no wrapping                       ◀── NEW
```

The canonical body in `safessh-skill/src/content/safessh.md` is unchanged. Adapter scope is wrapping only.

### 4.2 New adapters

#### 4.2.1 `cursor`

- **Filename:** `safessh.md`
- **Default install path:**
  - `Scope::User`: not supported (Cursor's user-scope rules live in editor settings, not on disk)
  - `Scope::Project`: `<cwd>/.cursor/rules/safessh.md`
- **Detection rule:** `<cwd>/.cursor` exists.
- **Format:**
  ```markdown
  ---
  description: SSH proxy for running gated commands on user-configured servers without seeing credentials. Use when the user asks to run commands on a remote server they've configured in safessh.
  globs:
  alwaysApply: false
  ---

  <canonical body>
  ```
  `globs:` left empty (the rule isn't path-scoped). `alwaysApply: false` matches Cursor's default for non-pinned rules.

#### 4.2.2 `gemini-cli`

- **Filename:** `GEMINI.md`
- **Default install paths:**
  - `Scope::User`: `~/.gemini/GEMINI.md`
  - `Scope::Project`: `<cwd>/GEMINI.md`
- **Detection rule:** `~/.gemini` exists (user) or `<cwd>/GEMINI.md` is present (project).
- **Format:** `## safessh\n\n<canonical body>\n` — same section-style approach as `agents-md`. Section editor logic is shared (see §4.4).

#### 4.2.3 `codex`

- **Filename:** `AGENTS.md`
- **Default install paths:**
  - `Scope::User`: `~/.codex/AGENTS.md`
  - `Scope::Project`: not added — project-scope is already covered by `Target::AgentsMd` writing to `<cwd>/AGENTS.md`. Adding a redundant codex-project entry would double-write.
- **Detection rule:** `~/.codex` exists.
- **Format:** `## safessh\n\n<canonical body>\n` — same as agents-md.

#### 4.2.4 `plain`

- **Filename:** caller-provided (no default).
- **Default install paths:** none — `--path <PATH>` is required.
- **Detection rule:** never auto-detected.
- **Format:** the canonical body verbatim, no frontmatter, no section header. Trailing newline added.

### 4.3 `--target all` and supported (target, scope) matrix

```
                User       Project
ClaudeCode      ✓          ✓
AgentsMd        —          ✓
Cursor          —          ✓
GeminiCli       ✓          ✓
Codex           ✓          —
Plain           —          —     (always requires --path)
```

`safessh skill install --target all --scope <user|project>` walks this matrix and installs to every supported pair at the chosen scope. Unsupported pairs are skipped with a one-line stderr note (`safessh: skill: skipping cursor (no user-scope install path)`). Plain is never included in `--target all` (it requires `--path`).

`safessh skill install --target all` without `--scope` defaults to `--scope user`; per-target user-scope misses are logged. (We considered `--scope both` but YAGNI for v0.6 — users who want both run the command twice.)

### 4.4 `safessh skill update`

Re-renders the embedded body and rewrites every detected installed copy.

```
$ safessh skill update [--dry-run] [--target <name>...] [--scope user|project|both]
```

- **Default behavior:** scan for installed copies via `detection::detect` + `default_path` for every (target, scope) pair, plus any caller-recorded custom paths (none in v0.6 — users with `--path` installs run `skill update --path <PATH>`). For each candidate path, **only rewrite if the file (or section, for section-style targets) is actually present** — non-installed locations are silently skipped, never created. Exit 0 on success even if zero copies found (with a stderr note: `safessh: skill: no installed copies found`).
- **`--dry-run`:** for each detected copy, compute the new body, compare to the current bytes, print a unified diff (or `(unchanged)`) to stdout. No writes. Exit 0 on success.
- **`--target` filter:** restrict to specific targets. Multiple `--target` flags accepted: `--target claude-code --target cursor`.
- **`--scope`:** restrict to a scope. `both` (the default) means user *and* project.
- **Section-style targets** (`agents-md`, `gemini-cli`, `codex`): the shared `install_md_section(path, "## safessh", body)` helper from §4.5 strips and rewrites only the safessh section; surrounding content is preserved. This is the existing `install_agents_md_section` logic, lifted into a shared helper.

### 4.5 Refactor: shared markdown-section helper

The existing `install_agents_md_section` and `strip_safessh_section` in `safessh-skill/src/install.rs` are AGENTS.md-specific by name only. Their behavior is "find and replace `## safessh` section in a markdown file." v0.6 extracts these into a shared helper used by `agents-md`, `gemini-cli`, and `codex`:

```rust
// safessh-skill/src/sections.rs
pub fn install_md_section(path: &Path, body: &str) -> Result<()>;
pub fn uninstall_md_section(path: &Path) -> Result<()>;
fn strip_section(content: &str) -> String;
```

The section header is a constant: `"## safessh"`. No public API removal; `install_to(Target::AgentsMd, ...)` keeps the same signature and now delegates to the shared helper.

### 4.6 `safessh skill detect`

New subcommand that prints the detection result for the current cwd + `$HOME`. Useful for diagnosing why `--target all` skipped something. Output format:

```
$ safessh skill detect
target        scope    path                                    installed
claude-code   user     ~/.claude/skills/safessh.md             yes
claude-code   project  ./.claude/skills/safessh.md             no
agents-md     project  ./AGENTS.md                             yes (section present)
cursor        project  ./.cursor/rules/safessh.md              detected, not installed
gemini-cli    user     ~/.gemini/GEMINI.md                     not detected
gemini-cli    project  ./GEMINI.md                             detected, not installed
codex         user     ~/.codex/AGENTS.md                      detected, section present
plain         —        —                                       requires --path
```

`--format json` is also supported (LLM-friendly). Exit 0 always.

### 4.7 Skill markdown updates (v0.6)

The canonical `safessh.md` is updated to:

- Document the new exit codes and CLI surface introduced in v0.5 (`audit query --since/--until/--decision/--exit-code/--target/--limit/--format`).
- Document `safessh skill install/update/detect` and `--target all` semantics.
- Add a "Audit log access" section pointing the agent at `safessh audit query` (not direct file reads).

This is a content edit, not a structural one. Adapter wrappers stay generic.

### 4.8 Testing strategy (v0.6)

- **Adapter unit tests** (`safessh-skill/tests/adapters.rs`) — already snapshotted. Add four new snapshots, one per new adapter.
- **Install / uninstall round-trip** for each new target: write to a temp dir, install, verify file content + permissions, uninstall, verify cleanup.
- **Section helper tests** for `install_md_section` / `uninstall_md_section`: untouched-file preservation, repeated install (idempotency), uninstall on a file with multiple sections (`## safessh` removed, neighbours intact).
- **Detection tests** for each target: positive case (directory exists), negative case (doesn't exist), case where the file exists but the directory hint doesn't.
- **`--target all` end-to-end** in a tempdir-backed `$HOME` and `$PWD`: installs to every supported pair, verifies skipped-pair logging.
- **`skill update --dry-run`** test: render expected diff for one outdated copy, assert no file mutation.

### 4.9 Rollout / migration

- v0.6 is purely additive — no existing installed skill files change format. `safessh skill update` after upgrade is recommended (announced in CHANGELOG); it idempotently rewrites with the freshest body, picking up the v0.5 audit-CLI documentation.
- The new targets' install paths are all under user-controlled directories. We **do not** create `~/.gemini/`, `~/.codex/`, or `~/.cursor/` automatically — if the directory doesn't exist, install fails with a clear "framework not installed" error and exit 33.

## 5. Cross-cutting

### 5.1 Workspace + dependencies

New crates to add as workspace dependencies:

- `rusqlite` (with `bundled` feature) — pins SQLite version, no system dep.
- `refinery` — embedded migrations.
- `humantime` — duration parsing (`7d`, `24h`).

These get used only inside `safessh-audit` (no leakage into `safessh-core`, `safessh-policy`, etc.). Workspace `Cargo.toml` updated.

The `safessh-skill` crate gains no new dependencies; the new adapters reuse existing helpers.

### 5.2 Exit codes

No new exit codes for either milestone:
- `audit query` argument errors map to existing exit `30` (usage).
- `skill install/update/detect` framework-missing errors map to existing exit `33` (skill install failure).
- SQLite catastrophic failures degrade to log-scan (warning on stderr) and return exit `0` if results were produced.

`docs/cli-reference.md` updated to document the new flags + how degraded modes signal themselves.

### 5.3 Safety invariant audit

| Invariant                                          | v0.5 impact                                                 | v0.6 impact         |
|----------------------------------------------------|-------------------------------------------------------------|---------------------|
| 1. Default-deny on parse failure                   | n/a                                                         | n/a                 |
| 2. Deny > Allow ordering                           | n/a                                                         | n/a                 |
| 3. Secrets never in argv                           | n/a — SQLite path is static                                 | n/a                 |
| 4. Audit-write before user-visible output          | **preserved.** SQLite catch-up is read-side only            | n/a                 |
| 5. Atomic file writes                              | DB writes via SQLite txn (atomic by default); DDL is one-shot in a tx | new adapters use existing `atomic::write_string` |
| 6. No outbound network                             | n/a — local SQLite only                                     | **preserved.** New skill content is `include_str!`'d; no fetches in `skill update` |
| 7. `--yolo` only bypasses policy                   | n/a                                                         | n/a                 |

A comment line `// SAFETY-INVARIANT-4: index is read-side only; never on the audit-write path.` is added at the top of `safessh-audit/src/sqlite/mod.rs`.

### 5.4 Cross-platform notes

- **macOS / Linux:** primary platforms. `rusqlite bundled` ships SQLite, no system dep.
- **Windows:** v0.5/v0.6 do not block on Windows; `~/.gemini`, `~/.codex` paths use `dirs::home_dir()` which already works there. SQLite tested on macOS + Linux CI; Windows considered best-effort (matching v0.4 status).

### 5.5 Out of scope (deferred)

- **Indexing across rotated logs.** v0.5 indexes only the live `audit.log`. Rotated `audit.log.<ts>` files are queryable via `--grep` fallback, not indexed. A future v0.5.1 may index them.
- **`audit query --output csv`.** Considered, dropped. JSONL + table covers the current need.
- **`skill install --scope both`.** YAGNI; the surface gets cluttered fast.
- **Cursor user-scope rules in editor settings.** Cursor's user-scope rules live in `~/Library/Application Support/Cursor/User/settings.json` and editing it is editor-managed; we don't touch it. Documented limitation.
- **Skill auto-update on safessh upgrade.** No. Update is opt-in (`safessh skill update`). SAFETY-INVARIANT-6 plus user-control principle.

## 6. Implementation order

The plan that consumes this spec implements in this order, all on a single `feature/v0.5-v0.6` branch:

1. v0.5 SQLite schema + migrations + `Index::open_or_create` + catch-up (audit crate, no CLI surface yet).
2. v0.5 `Index::query` + filter struct + integration into `safessh-cli/src/commands/audit.rs`.
3. v0.5 `audit query` flag additions + table/jsonl/count formatters.
4. v0.5 TUI audit screen rewire to SQLite-backed query.
5. v0.5 `docs/audit.md`.
6. v0.6 `Target` enum extension + section helper refactor + four new adapters.
7. v0.6 `skill install --target all` + `skill update` + `skill detect`.
8. v0.6 skill markdown content update for v0.5 features.
9. Docs pass: `cli-reference.md`, `skill.md`, `README.md` features table.
10. Release rehearsal: v0.5 release first (audit power is shippable on its own), then v0.6 (skill targets shippable on its own). Each gets its own tag and CHANGELOG entry.

The release rehearsal step splits the bundled feature branch into two tagged releases on `main` — `v0.5.0` and `v0.6.0` — back-to-back. Each tag triggers cargo-dist independently, so users get two CHANGELOG entries and can pin to either.
